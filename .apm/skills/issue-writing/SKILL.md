# issue の書き方

要望・不具合・タスクから、**対応に着手できる** issue のタイトルと本文を書く。この skill は、特定の課題管理ツール（Backlog / GitHub Issues / Jira 等）に依存せず、文面の作成だけに専念する。どのツールに貼り付けても通用する、過不足のない文章を目指す。

> 本文の文章は `stop-ai-slop-jp` skill の規範（立場・論証・語彙・整形）に従って書く。

## 大原則

課題の正体は**作業の依頼書**だ。思いつきのメモではない。これを読んだ別の人（や未来の自分）が、何をすれば「終わり」なのかを判断して着手できる状態にする。

着手できない課題には、共通して同じ穴がある。何が問題なのか（現状）と、どうなれば解決なのか（あるべき姿・完了条件）が書かれていない。AI に丸投げすると「○○を改善する」のような願望だけが並び、誰も「では何をすれば閉じられるのか」を判断できない。

書き始める前に、これに答えられるか確かめる。

- 何が問題か。なぜ今これに対応する必要があるのか（背景）
- どうなれば、この課題を閉じてよいのか（完了条件）
- 不具合なら、再現手順と「期待する挙動／実際の挙動」を出せるか

答えられないなら、本文を埋める前にユーザーに問う。再現手順や完了条件を推測ででっち上げない。

## コアルール

### A. 中身（最優先）

1. **現状と完了条件をセットで書く**。「何が問題か」と「どうなれば解決か（受け入れ条件）」の両方を書く。願望だけで終わらせない。
2. **1 課題 = 1 関心事**。複数の課題を1つに詰め込まない。大きすぎるなら分割を提案する。
3. **タイトルは具体的に**。「バグ修正」ではなく「どこで・何が起きるか」が分かる主語付きで、簡潔に。種別を一目で示したいなら `[Bug]` / `[Feature]` / `[Task]` のような接頭辞を付けてもよい（ツール側に種別欄があるなら任意）。
4. **事実と要望を分ける**。観測された事実（ログ・エラー・再現手順）と、こうしたいという要望・提案を混ぜない。
5. **推測で埋めない**。再現手順・原因・完了条件が不明なら、空欄にせずユーザーに確認する。

### B. 種別ごとの要点

6. **不具合（Bug）**: 再現手順を番号付きで、期待する挙動・実際の挙動・環境（OS / ブラウザ / バージョン等）を書く。再現できない情報は起票しても直せない。
7. **要望・機能（Feature）**: 解決したい課題（なぜ要るか）を先に書き、提案する解決はその後に置く。実装の押し付けにしない。
8. **タスク（Task）**: 作業のゴールと完了条件（チェックボックス）を書く。関連する調査・設計の文脈をリンクする。

### C. 形式

9. **TEMPLATE に従う**（`references/TEMPLATE.md`）。種別に合わせて使い、該当しないセクションは省く。空見出しを残さない。
10. **関連を紐付ける**。関連する課題 / PR / DesignDoc / ADR があればリンクや参照を書く。
11. **飾りと冗長を削る**。意味のない前置き、同じことの言い換え、飾りの絵文字を入れない。

## 入力ソース

| 優先 | ソース                       | 取得方法                                             |
| ---- | ---------------------------- | ---------------------------------------------------- |
| 1    | ユーザーが伝えた要望・不具合 | 背景・やりたいこと・症状をそのまま使う               |
| 2    | 観測された事実               | エラーログ、再現手順、スクリーンショット、該当コード |
| 3    | 既存の文脈                   | 関連する既存課題、`docs/design/`・`docs/adr/`        |

複数あれば統合する。矛盾や不足は推測で埋めず、ユーザーに確認する。

## 出力

- 課題の**タイトル**と**本文**をドラフトとして提示する。
- そのまま課題管理ツールに貼り付けられる形にする。起票（どのツールに、どの種別／優先度で登録するか）はユーザーの作業に委ねる。
- ファイルとしては保存しない。

## フォーマット

`references/TEMPLATE.md` をテンプレートに使う。粒度は下表で判断する（表が正、TEMPLATE のコメントは記入ヒント）。

| セクション           | 書くこと                                   | 書かないこと                 |
| -------------------- | ------------------------------------------ | ---------------------------- |
| 背景・目的           | 何が問題か・なぜ今対応するか（1〜2段落）   | 解決策の決め打ち             |
| 現状（不具合の場合） | 再現手順・期待する挙動・実際の挙動・環境   | 推測した原因の断定           |
| 完了条件             | どうなれば閉じてよいか（チェックボックス） | 願望（「改善する」だけ）     |
| 提案・補足           | 想定される解決の方向性、調査メモ（あれば） | 確定していない設計の押し付け |
| 関連資料             | 関連する課題 / PR / DesignDoc / ADR        | リンクのない作業ログ         |

## 手順

1. **入力を確認する** — 要望か不具合かタスクか、背景・事実がそろっているかを把握する。曖昧なら、種別・背景・完了条件を先に問う。
2. **種別とスコープを決める** — 1 課題 = 1 関心事に絞る。大きすぎるなら分割を提案する。
3. **事実を集める** — 不具合なら再現手順・環境・ログ。必要なら関連する既存課題・`docs/` を確認する。
4. **タイトルと本文を起草する** — 現状と完了条件をセットで書く。種別に合わせて TEMPLATE を使う。
5. **クイックチェックで見直す** — 下記。満たせない項目があれば書き直す。
6. **ドラフトを提示する** — タイトル・本文を見せる。起票はユーザーが使う課題管理ツールで行う。
7. **次ステップを伝える** — 必要なら design-doc skill で設計の認識合わせに進めることを案内する。

## クイックチェック

提示前に、着手する担当者になったつもりで上から確認する。

### 中身
- [ ] 現状（何が問題か）と完了条件（どうなれば解決か）が両方あるか?
- [ ] 1 課題が1関心事に絞れているか?
- [ ] タイトルが具体的で、何の課題か一目で分かるか?
- [ ] 事実と要望・推測を分けて書いているか?
- [ ] 不明点を推測で埋めず、確認に回しているか?

### 種別ごと
- [ ] 不具合なら、再現手順・期待／実際の挙動・環境があるか?
- [ ] 要望なら、解決策より先に「解決したい課題」を書いているか?
- [ ] タスクなら、完了条件がチェックボックスで書けているか?

### 形式
- [ ] 関連する課題 / PR / DesignDoc / ADR を紐付けているか?
- [ ] 該当しない空見出しを残していないか?
- [ ] 飾りの絵文字・冗長な言い換え・空の前置きがないか?

迷ったら優先順位は上から。現状と完了条件が空なら、文章を整えても着手できる依頼書にはならない。

## やらないこと

- 「○○を改善する」のような願望だけで、完了条件のない課題を出力する
- 再現手順・環境のない不具合を起票できる体で書く
- 観測事実と推測した原因を混ぜて、断定として書く
- 複数の独立した課題を1つの課題に詰め込む
- 特定のツール（Backlog / GitHub / Jira 等）の操作・起票手順に踏み込む（この skill は文面に専念する）
