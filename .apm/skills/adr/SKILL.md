# ADR 作成

認識合わせの済んだ DesignDoc（または合意済みの議論・決定）から、**確定した設計判断**を ADR として `docs/adr/` に記録する。

> 本文の文章は `stop-ai-slop-jp` skill の規範（立場・論証・語彙・整形）に従って書く。

## 大原則

ADR の正体は「**なぜその判断をしたか**」の記録だ。半年後に「なんでこうなってるんだっけ」と思った未来の自分やチームへの手紙になる。

だから本体は「決定」ではなく、その手前の**コンテキスト**（どんな制約があり、なぜこの判断が要ったか）と、**結果**（何を得て、何を捨てたか）にある。AI に書かせると、ここが逆になる。「決定」だけが立派に書かれ、なぜ必要だったか・何を捨てたかが空になる。それでは未来の読み手の問いに答えられない。

書く前に、これに答えられるか確かめる。

- この判断は本当に**確定**しているか（検討中の迷いを決定として書いていないか）
- なぜこの判断が必要だったか、**具体的な制約**を1つ挙げられるか
- 採用しなかった案と、引き受けた**トレードオフ**は何か

## コアルール

### A. 立場と中身（最優先）

1. **1 ADR = 1 判断**。DesignDoc に判断が複数あれば、それぞれ別 ADR にする。詰め込まない。
2. **確定した判断だけ書く**。ここでの「確定」は、その案でいくと**チームで合意済み**かを指す。未合意（懸念事項）は ADR にせず DesignDoc に残す。なお「合意済みか」と、ステータスの `提案中`/`承認`（formal な承認の有無）は**別の軸**（決め方は「フォーマット」）。
3. **決定は断定形で言い切る**。「〜する」。「〜する方向で検討」のような逃げを残さない。
4. **両論併記しない**。選ばなかった案は「結果」のトレードオフで触れる。決定そのものは1つ。
5. **コンテキストは具体に降りる**。「保守性のため」で止めず、このプロジェクトのどの制約・事実が判断を要求したかを書く。
6. **都合の悪いことも書く**。結果にトレードオフ・残課題を正直に書く。良い面だけ並べない。

### B. 構造と追跡性

7. **連番と命名規則を守る**（下記）。
8. **相互リンクを張る**。ADR から元 DesignDoc へ、元 DesignDoc から作成した ADR へ、双方向に。

### C. 形式

9. **TEMPLATE に従う**。コンテキスト → 決定 → 結果 の順で簡潔に。1 ADR は1判断ぶんなので短く収まるはず。長くなったら、判断が複数混ざっていないか（コアルール 1）を疑う。
10. **飾りと冗長を削る**。空の前置き・同義の言い換えを入れない。

## 入力

- 主たる入力は対象の DesignDoc（`docs/design/{スラッグ}.md`）。指定がなければどれを ADR 化するか確認する。
- DesignDoc がない確定判断（インシデント対応・スパイク・口頭合意など）も対象にしてよい。その場合は根拠にした議論・事実の出どころを「関連資料」に残す。
- いずれの場合も、対象は**合意済み**の判断だけ。未合意（懸念事項）は ADR にしない。

## 出力先

- `docs/adr/{連番}-{スラッグ}.md`（例: `0001-async-with-sqs.md`）
- 連番は既存 ADR の最大値 +1。**4桁ゼロ埋め**。ブランチ並行作業で番号が衝突しうるので、保存直前に `docs/adr/` の最新を再確認する
- 本文冒頭の見出し `# ADR-{連番}: タイトル` もファイル名の連番に合わせる
- 既存ファイルがあれば上書きせず、ユーザーに確認する
- この skill は**新規作成専用**。既存 ADR の `廃止` / `置換` は通常の編集で行う

## フォーマット

`references/TEMPLATE.md` に従う。

**ステータスの決め方**（取り違えやすいので先に）。ADR にするのは「その案でいく」と合意済みの判断だけ（コアルール 2）。そのうえで `ステータス` は別軸で、**formal な承認の有無**を表す。

- 合意済み・正式承認も済み → `承認`
- 合意済み・正式承認はこれから → `提案中`
- 未合意（懸念事項） → ADR にしない。DesignDoc に残す
- `廃止` / `置換` は既存 ADR の更新時のみ（この skill の対象外）

| セクション   | 書くこと                                                             |
| ------------ | -------------------------------------------------------------------- |
| ステータス   | 上の決め方に従い `提案中` / `承認`                                   |
| コンテキスト | なぜこの判断が必要か。**具体的な**制約・前提（DesignDoc の背景から） |
| 決定         | 採用した案を**断定形**で（「〜する」）                               |
| 結果         | 良い影響・**捨てたトレードオフ**・残課題                             |
| 関連資料     | 元の DesignDoc、関連する課題 / PR / ADR                              |

## 手順

1. **対象の DesignDoc を特定する** — 未指定なら確認し、中身を読む。
2. **確定した判断を抽出する** — 合意済みのものだけ。未確定（懸念事項）は除外する。
3. **ステータスを決める** — 「フォーマット」の決め方に従う（合意済みでも正式承認前なら `提案中`）。判断できないときだけユーザーに確認する。
4. **判断ごとに ADR を起こす** — 1判断1ファイル。連番を採番し、見出し `# ADR-{連番}` も揃える。
5. **コンテキストと結果を埋める** — なぜ必要か・何を捨てたかを具体に書く。
6. **クイックチェックで見直す** — 下記。満たせない項目があれば書き直す。
7. **相互リンクを張って保存する** — `docs/adr/{連番}-{スラッグ}.md`。元 DesignDoc にも作成した ADR を追記する。

## クイックチェック

保存前に上から確認する。

### 立場と中身
- [ ] 1 ADR が1判断に絞れているか?
- [ ] 確定した判断だけを書いているか?（検討中の迷いを決定にしていないか）
- [ ] 決定を断定形で言い切っているか?（両論併記なし）
- [ ] コンテキストが一般論でなく、このプロジェクトの具体的制約に降りているか?
- [ ] 結果に捨てたトレードオフ・残課題を正直に書いているか?

### 構造と形式
- [ ] ステータスを「決め方」に沿って設定したか?（合意済みでも正式承認前なら提案中）
- [ ] 連番・命名（4桁ゼロ埋め）と見出しが揃っているか?
- [ ] DesignDoc と双方向にリンクしているか?
- [ ] 飾り・冗長を削れているか?

迷ったら優先順位は上から。コンテキストとトレードオフが空なら、決定だけ立派でも記録の意味がない。

## やらないこと

- 未確定の判断を「決定」として書く
- 1 ADR に複数の独立した判断を詰め込む
- 良い面だけ書き、捨てたトレードオフを隠す
- 相互リンク（DesignDoc ↔ ADR）を張らずに保存する
- 連番を重複・欠番させる
