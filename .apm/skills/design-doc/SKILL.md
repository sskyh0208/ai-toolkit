# DesignDoc 作成

要件・課題から、チームの認識合わせ用 DesignDoc を `docs/design/` に作る。

> 本文の文章は `stop-ai-slop-jp` skill の規範（立場・論証・語彙・整形）に従って書く。

## 大原則

DesignDoc の正体は**認識合わせ**だ。完璧な設計書を作ることではない。これは実装前に「自分たちは何を、なぜそう作るのか」をチームで握るための叩き台になる。

AI に丸投げすると、ここが抜ける。テーマだけ渡すと、どこかで読んだような一般論の詰め合わせが出てくる。「保守性が向上する」「ユーザー体験が改善される」と書いてあっても、誰も「いや、Bだろう」と反論できない。反論できない設計案はレビューできない。

書き始める前に、これに答えられるか確かめる。

- 自分（チーム）は何を作ると**引き受ける**のか
- 主要な設計判断は何で、**なぜそれを選ぶ**のか
- 何がまだ決まっていなくて、認識合わせで**何を決めたい**のか

答えられないなら、テンプレートを埋める前に、ソースを読み直すかユーザーに問う。空のテンプレートは作らない。

## コアルール

### A. 立場と中身（最優先）

1. **何を作るか引き受ける**。「〜できるとよい」「〜が望ましい」のような願望で逃げない。「〜を作る」と書く。
2. **設計判断は根拠とセットで書く**。「○○を採用する」だけにせず「△△なので○○にする」と理由を添える。ただし DesignDoc は叩き台なので断定しすぎない。確定は ADR でやる。
3. **両論併記で終わらせない**。「A もあり B もある」は判断の放棄。方向性を1つ示し、迷いは「懸念事項」に回す。
4. **このプロジェクトの事実に降りる**。関連コード・既存 ADR・関連課題を読み、根拠にできる事実を拾う。一般論のテンプレ埋めをしない。
5. **推測で埋めない**。ソース間の矛盾や不明点はユーザーに確認する。

### B. スコープと構造

6. **スコープを1つに絞る**。1 DesignDoc = 1 テーマ。
7. **未決定は「懸念事項」へ**。隠さず、「決めたい問い」の形で書く。
8. **詳細設計に踏み込まない**。実装手順・クラス名の列挙・API 仕様の確定はやらない。

### C. 形式

9. **TEMPLATE に従う**（下記）。
10. **200行以下**に収める（コメント・区切りを除く本文の目安。超えたら詳細に踏み込みすぎを疑い、削るか別ドキュメント化を提案する）。
11. **装飾と冗長を削る**。意味のない前置き、同じことの言い換え、飾りの絵文字は入れない。

## 入力ソース

| 優先 | ソース               | 取得方法                                       |
| ---- | -------------------- | ---------------------------------------------- |
| 1    | ユーザーが伝えた要件 | 背景・やりたいこと・制約をそのまま使う         |
| 2    | 関連する課題         | 課題の本文・コメントから背景・要望・経緯を読む |
| 3    | 既存ドキュメント     | 関連 ADR、README、調査メモ等                   |

複数あれば統合する。矛盾は推測で埋めず、ユーザーに確認する。

## 出力先

- `docs/design/{スラッグ}.md`（例: `profile-export.md`）。スラッグは英小文字・ハイフン区切り（kebab-case）
- 既存ファイルがあれば上書きせず、ユーザーに確認する

## フォーマット

`references/TEMPLATE.md` をテンプレートに使う。粒度は下表で判断する（表が正、TEMPLATE のコメントは記入ヒント）。

| セクション           | 書くこと                                     | 書かないこと                  |
| -------------------- | -------------------------------------------- | ----------------------------- |
| 背景・目的           | なぜ今やるか・何を解決するか（1〜2段落）     | 実装手順、クラス名の列挙      |
| 概要                 | 作るもの・スコープ（箇条書き）               | 詳細な API 仕様、画面ワイヤー |
| 主要な設計判断       | 技術選定・構成の**方向性と根拠**             | 確定していない詳細設計        |
| 懸念事項・未決定事項 | 認識合わせで決めたい問い（チェックボックス） | 解決済みの細部                |
| 関連資料             | 関連する課題 / ADR / PR のリンク             | リンクのない作業ログ          |

- 主要な設計判断が複数あるときは `### 2.` `### 3.` と番号を増やす

## 手順

1. **入力を確認する** — 要件・関連課題・既存ドキュメントの有無を把握。ソースも対象も曖昧なら、先に問う（テーマ / 要件のソース / 主な変更領域 backend・frontend・infra）。
2. **スコープを1つに絞る** — テーマ・対象領域・出力スラッグを決める。
3. **ソースを読む** — 関連課題、`docs/`、必要ならコードを横断確認する。
4. **テンプレートに沿って起草する** — 見出しは `# DesignDoc: {テーマ}`。立場と根拠を入れる。未決定は「懸念事項」へ回す。
5. **クイックチェックで見直す** — 下記。満たせない項目があれば書き直す。
6. **保存する** — `docs/design/{スラッグ}.md` に書き出す。
7. **次ステップを伝える** — 認識合わせが必要なこと、合意後に `adr` skill で確定判断を ADR 化できることを案内する。

## クイックチェック

保存前に上から確認する。

### 立場と中身
- [ ] 「何を作るか」を引き受けて言い切っているか?（願望で終わっていないか）
- [ ] 主要な設計判断に「なぜそれを選ぶか」の根拠があるか?
- [ ] 両論併記で判断を放棄していないか?
- [ ] 一般論でなく、このプロジェクト固有の制約・コードに降りているか?
- [ ] 不明点を推測で埋めていないか?

### スコープと構造
- [ ] テーマが1つに絞れているか?
- [ ] 未決定が「決めたい問い」として懸念事項に書けているか?
- [ ] 詳細設計（実装手順・クラス名）に踏み込みすぎていないか?

### 形式
- [ ] 200行以下か?
- [ ] 空見出し（懸念事項・関連資料など）を残していないか?
- [ ] 飾りの絵文字・冗長な言い換え・空の前置きがないか?

迷ったら優先順位は上から。立場が立っていないのに形式だけ整えても、レビューできる叩き台にはならない。

## やらないこと

- ソースも要件もなく、テンプレートだけ埋めたファイルを出力する
- 根拠のない技術選定を確定事項として書く
- 認識合わせが済む前に ADR 作成・タスク分割へ進む
- 200行を超える詳細設計
