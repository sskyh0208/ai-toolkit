# PR の書き方

ブランチの変更内容から、**レビューしやすい** PR のタイトルと本文を書く。この skill は文面に専念する。GitHub への作成は `github` skill が担う。

> 本文の文章は `stop-ai-slop-jp` skill の規範（立場・論証・語彙・整形）に従って書く。

## 大原則

PR の正体は**レビュー依頼**だ。きれいな説明文を書くことではない。レビュアーが「この変更を取り込んでよいか」を最小の労力で判断できるようにするための材料を渡す。

差分そのものは「何を変えたか（what）」を語る。だが差分を読んでも「**なぜ**変えたか」「**どう確認したか**」「**どこを重点的に見てほしいか**」は分からない。AI に丸投げすると、ここが抜ける。差分を箇条書きで言い換えただけの、読んでも判断材料にならない本文が出てくる。

書き始める前に、これに答えられるか確かめる。

- この変更は、何を解決するために入れるのか（背景・目的）
- レビュアーは何を確認すれば、安心して承認できるのか（確認方法・観点）
- この変更で何を引き受け、何を後回し・トレードオフにしたのか

答えられないなら、本文を埋める前に diff・コミット・関連する課題を読み直す。実在しない確認手順や根拠を推測で書かない。

## コアルール

### A. 中身（最優先）

1. **レビュアー視点で書く**。差分を言い換えるのではなく、「なぜこの変更が要るか」「何を確認すれば承認できるか」を書く。差分を読めば分かることの再掲に紙幅を使わない。
2. **1 PR = 1 目的**。無関係な変更を混ぜない。レビューできない巨大 PR になりそうなら、分割をユーザーに提案する。
3. **タイトルは commit prefix 規約に従う**。`feat:` `fix:` `docs:` `style:` `refactor:` `perf:` `test:` `chore:` のいずれかを付け、日本語・命令形・50字以内・末尾ピリオドなし。
4. **変更を実際の事実から拾う**。`git diff`・`git log` を読み、起きている変更に基づいて書く。やっていないことを書かない。
5. **関連を紐付ける**。関連する課題 / DesignDoc / ADR をリンクする。課題の識別子と URL を本文に書く。課題管理ツールが GitHub Issues でなければ `Closes #N` では課題は閉じないので、クローズは課題管理ツール側で行う。

### B. 確認とリスク

6. **確認方法を書く**。レビュアーと自分が再現できる形で、テスト手順・動作確認の手順と結果を書く。「確認済み」だけで終わらせない。
7. **レビューで見てほしい点を書く**。迷った設計判断・自信のない箇所・トレードオフを正直に挙げる。問題を隠さない。
8. **影響範囲とリスクを明示する**。破壊的変更・マイグレーション・設定変更・依存追加があれば、見落とされない位置に書く。

### C. 形式

9. **テンプレートに従う**。リポジトリに PR テンプレートがあればそれを使い、なければ `references/TEMPLATE.md` を使う（→「フォーマット」）。該当しないセクションは省く。空見出しを残さない。
10. **base ブランチと draft 可否を決める**。マージ先がデフォルトブランチでよいか、まだ未完成で draft にすべきかを確かめる（実際の作成時に `github` skill へ渡す）。
11. **飾りと冗長を削る**。意味のない前置き、差分の単純な言い換え、飾りの絵文字を入れない。

## 入力ソース

| 優先 | ソース         | 取得方法                                                       |
| ---- | -------------- | -------------------------------------------------------------- |
| 1    | ブランチの変更 | `git diff <base>...HEAD`、`git log <base>..HEAD`、`git status` |
| 2    | 関連する課題   | 課題の本文・コメントから背景・受け入れ条件を読む               |
| 3    | 設計の文脈     | 関連 DesignDoc（`docs/design/`）、ADR（`docs/adr/`）           |

複数あれば統合する。矛盾は推測で埋めず、ユーザーに確認する。

## 出力

- PR の**タイトル**と**本文**、および base ブランチ・draft 可否をドラフトとして提示する。
- 作成するなら `github` skill に渡す（push の確認・PR 作成はそちらが担う）。
- ファイルとして保存はしない（PR 本文は GitHub 上が正）。

## フォーマット

**まずリポジトリの PR テンプレートを探し、あればそれに従う**。次の場所を順に見て、最初に見つかったものを使う。

1. `.github/PULL_REQUEST_TEMPLATE.md`
2. `.github/PULL_REQUEST_TEMPLATE/`（複数テンプレート。用途に合うものを選ぶ）
3. リポジトリルートの `PULL_REQUEST_TEMPLATE.md`
4. `docs/PULL_REQUEST_TEMPLATE.md`

いずれも見つからない場合に限り、`references/TEMPLATE.md` を使う。どちらの場合も、中身の粒度は下表で判断する（表が正、テンプレートのコメントは記入ヒント）。リポジトリのテンプレートを使うときは、その見出し構成を尊重しつつ、下表の観点（なぜ・確認方法・レビュー観点・影響）が埋まるように書く。

| セクション         | 書くこと                                         | 書かないこと               |
| ------------------ | ------------------------------------------------ | -------------------------- |
| 概要               | この PR で何を・なぜやるか（1〜2文）             | 差分の逐次的な言い換え     |
| 変更内容           | レビュアーが理解すべき主要な変更（箇条書き）     | 全ファイルの機械的な列挙   |
| 背景・関連         | 解決する課題、関連する課題 / DesignDoc / ADR     | リンクのない作業ログ       |
| 確認方法           | 実行したテスト・動作確認の手順と結果             | 「確認済み」だけの一行     |
| レビュー観点・懸念 | 重点的に見てほしい点、トレードオフ、残課題       | 都合の良い話だけ           |
| 影響・リスク       | 破壊的変更・マイグレーション・設定変更（あれば） | 該当しないのに残す空見出し |

## 手順

1. **変更を把握する** — `git status`、`git diff <base>...HEAD`、`git log <base>..HEAD` で変更とコミットを読む。base が不明なら確認する。
2. **スコープを確かめる** — 1 PR = 1 目的か。無関係な変更が混ざっていないか。巨大すぎるなら分割を提案する。
3. **文脈を読む** — 関連する課題、`docs/design/`・`docs/adr/` を必要に応じて確認する。
4. **テンプレートを決める** — リポジトリの PR テンプレートを探し、あればそれを使う。なければ `references/TEMPLATE.md` を使う（→「フォーマット」）。
5. **タイトルと本文を起草する** — prefix 規約でタイトルを付け、決めたテンプレートに沿って本文を書く。なぜ・確認方法・観点を埋める。関連する課題の識別子・URL を本文に書く（クローズは課題管理ツール側で行う）。
6. **クイックチェックで見直す** — 下記。満たせない項目があれば書き直す。
7. **ドラフトを提示する** — タイトル・本文・base・draft 可否を見せる。作成するなら `github` skill に渡す（push・PR 作成はそちら）。

## クイックチェック

提示前に、レビュアーになったつもりで上から確認する。

### 中身
- [ ] 差分の言い換えでなく「なぜ変えたか」が書けているか?
- [ ] 1 PR が1目的に絞れているか?
- [ ] タイトルが prefix 規約・50字以内・命令形・ピリオドなしか?
- [ ] 実際の diff・コミットに基づき、やっていないことを書いていないか?
- [ ] 関連する課題 / DesignDoc / ADR を紐付けているか?

### 確認とリスク
- [ ] 確認方法が再現できる形で書いてあるか?
- [ ] 見てほしい点・トレードオフ・残課題を正直に書いているか?
- [ ] 破壊的変更・マイグレーション・依存追加を明示しているか?

### 形式
- [ ] 該当しない空見出しを残していないか?
- [ ] 飾りの絵文字・冗長な言い換え・空の前置きがないか?

迷ったら優先順位は上から。確認方法と観点が空なら、説明だけ整えてもレビューは進まない。

## やらないこと

- 差分を箇条書きで言い換えただけの、判断材料にならない本文を出力する
- 実行していないテスト・確認手順を「確認済み」と書く
- 無関係な変更を1つの PR に詰め込む
- 破壊的変更・マイグレーションを目立たない位置に埋める
- 自分で GitHub に PR を作成する（作成は github skill が、ユーザー確認のうえで行う）
