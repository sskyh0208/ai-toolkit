# コードレビュー

コード変更を、**直すべき箇所を見極められる指摘**に落とす。この skill はレビューの観点と重要度の基準を定める。

対象は PR に限らない。コミット前の作業差分、ステージ済みの変更、特定のブランチ、1ファイル、貼られたスニペットでも同じ観点で見る。GitHub PR への投稿が要るときだけ、`pull-request-review` skill がこの skill の観点を使って投稿手順を足す。

> 指摘文を書くときは `stop-ai-slop-jp` skill の規範（立場・論証・語彙・整形）に従う。

## 大原則

レビューの正体は**「この変更を取り込んでよいか」「直すなら何が優先か」の判断材料を渡すこと**だ。指摘の数を増やすことでも、網羅的に読んだ証拠を残すことでもない。

AI にやらせると、ここがずれる。差分を逐次なぞって「ここも直せる」と並べるだけになる。コードを書いた人が知りたいのは「今のままで安全か」「直すなら何からか」だ。指摘は重要度で絞り、取り込みを止めるべき指摘とそうでない指摘を混ぜない。

レビュー前に確かめる。

- この変更は何を解決しようとしているか
- 変更の中で**取り込む前に必ず直すべき**箇所はあるか
- 指摘は変更されたコードに限れているか

## 入力

レビュー対象は依頼の文脈から判断する。指定がなければ何を見ればよいか確認する。

| 対象                       | 取得方法                                         |
| -------------------------- | ------------------------------------------------ |
| 作業中の変更（未コミット） | `git diff`、`git status`                         |
| ステージ済みの変更         | `git diff --staged`                              |
| ブランチの変更             | `git diff <base>...HEAD`、`git log <base>..HEAD` |
| 特定ファイル               | ファイルを読む                                   |
| スニペット                 | 依頼に貼られた内容をそのまま使う                 |

PR のレビューは `pull-request-review` skill が対象取得（`gh pr view` / `gh pr diff`）と投稿まで担う。

## 重要度ラベル

すべての指摘に付ける。MUST と SHOULD を混ぜて並べない。

| ラベル         | 基準                                                           |
| -------------- | -------------------------------------------------------------- |
| 🔴 **MUST**     | バグ、セキュリティ問題、データ損失リスク。取り込む前に修正必須 |
| 🟡 **SHOULD**   | 規約違反、保守性の低下、パフォーマンス懸念。強く推奨           |
| 🟢 **NITS**     | より良い書き方の提案、軽微なスタイル。任意                     |
| 💬 **QUESTION** | 意図が不明な箇所、設計判断の確認                               |

## レビュー観点

観点ごとに詳細を `references/` に分けた。レビュー対象の性質に応じて、該当する観点のファイルを読んで適用する。

| 観点                     | 見るもの                                         | 参照                        |
| ------------------------ | ------------------------------------------------ | --------------------------- |
| 正確性・バグリスク       | ロジック・境界条件・非同期・エラー処理           | `references/correctness.md` |
| セキュリティ             | インジェクション・認証認可・機密情報・データ公開 | `references/security.md`    |
| プロジェクト規約・可読性 | `rules/` 準拠・命名・重複・複雑さ                | `references/conventions.md` |
| テスト                   | カバレッジ・正常系/異常系/境界値                 | `references/testing.md`     |
| パフォーマンス           | N+1・無駄な計算・メモリ・インデックス            | `references/performance.md` |
| 設計・スコープ           | 単一責務・変更の範囲・依存追加                   | `references/design.md`      |

すべての観点を毎回フルに当てる必要はない。変更がインフラなら correctness と security と conventions、UI コンポーネントなら conventions と design、というように、対象の性質に合うものを選ぶ。

## 進め方

1. **対象と意図を掴む** — 何をレビューするか（上記の入力）を確定する。そのうえで「なぜこの実装をしたのか」を、説明・コミット・関連 Backlog 課題・関連 DesignDoc から把握する。あわせて `docs/` 全般の決まり事、特に `docs/adr/` で明文化された確定事項（採用した方式・禁止した選択肢）を読み、変更がそれに反していないかを判断材料にする。**意図が掴めないままレビューを始めない**。説明・履歴・ドキュメントを当たっても「なぜそうしたか」が分からなければ、推測で良し悪しを判断せず、先に実装者へ意図を確認する。
2. **観点を選ぶ** — 対象の性質に応じて、当てる観点を決める。
3. **観点ファイルを読んで適用する** — 該当する `references/` を読み、変更箇所に当てて指摘を出す。
4. **重要度を付けて絞る** — 各指摘に MUST / SHOULD / NITS / QUESTION を付ける。同じパターンは代表1箇所にまとめ「同様の箇所が N 件」と添える。
5. **まとめて伝える** — MUST が1件以上なら「取り込む前に要修正」、なければ「致命的な問題なし」と判定を示し、指摘を重要度順に並べる。

## 出力

- 指摘を重要度順に、対象箇所（ファイル・行）とともに示す。
- 修正提案があれば、変更後のコードを添える。
- どこへ投稿・反映するかはこの skill の範囲外（ローカルレビューはその場で提示して終わり）。PR への投稿は `pull-request-review` に任せる。

## クイックチェック

- [ ] レビュー前に変更の意図（何を解決するか）を掴めているか?（掴めないなら推測せず実装者に確認）
- [ ] 変更されたコードだけを対象にしているか?（既存コードの粗探しをしていない）
- [ ] すべての指摘に重要度ラベルが付いているか?
- [ ] 同じパターンをまとめているか?
- [ ] 対象の性質に合った観点を当てているか?
- [ ] 指摘が「取り込んでよいか」の判断に役立つか?

## やらないこと

- 「なぜ実装したのか」が分からないまま、推測でレビューを進める
- `docs/`・`docs/adr/` の確定事項を確認せず、明文化済みの方針に反した指摘・見落としをする
- 重要度を付けずに指摘を並べる
- 変更されていない既存コードの問題を新たに指摘する
- 関係ない観点まで機械的に当てて、ノイズを増やす
- 差分を言い換えただけの、判断材料にならない指摘を出す
