# プロジェクト規約・可読性

既存のやり方に沿っているか。半年後に読んで分かるか。

## プロジェクト規約
- `rules/` 配下（`.cursor/rules/` / `.claude/rules/`）の該当ルールを読み、違反がないか確認する
- `docs/` 全般の決まり事、特に `docs/adr/` で明文化された確定事項（採用した方式・禁止した選択肢）に反していないか
- 既存の技術スタック・ディレクトリ構成・命名規則に従っているか
- 同じ目的の既存実装と作法が揃っているか（独自流儀を持ち込んでいないか）

## 確定事項との整合
- ADR で却下された選択肢を持ち込んでいないか
- ADR で決めた方式から外れる実装になっていないか（外れるなら、それは意図的か・新たな判断が要るか）

## 命名
- 名前が役割を表しているか（誤解を招く名前、省略しすぎ）
- 真偽値・関数・コレクションの命名規約に沿っているか

## 重複・構造
- コピペされた重複ロジック（共通化の余地）
- 1関数が長すぎ・責務過多でないか
- ネストが深すぎないか（早期 return で平坦にできるか）

## 可読性
- マジックナンバー・マジック文字列が定数化されているか
- 不要なコメント（コードの言い換え）・コメントアウトされた死蝋コード
- 意図が非自明な箇所に、なぜそうするかの説明があるか

## 重要度の目安
- 明確な規約違反 → 🟡 SHOULD
- 可読性・保守性を下げるが動作に影響なし → 🟢 NITS
- 好みの範囲、判断を仰ぎたい → 💬 QUESTION
