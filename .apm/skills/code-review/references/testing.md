# テスト

変更が壊れていないことを、テストが保証しているか。

## カバレッジ
- 変更したロジックに対応するテストが追加・更新されているか
- バグ修正に、再発を防ぐ回帰テストが付いているか
- 公開インターフェース・分岐・エラー経路が踏まれているか

## ケースの質
- 正常系だけでなく、異常系（失敗・例外）を検証しているか
- 境界値（0・空・最大・最小・null）を踏んでいるか
- 同値クラスの代表値を押さえているか（網羅でなく要点）

## テスト自体の健全性
- アサーションが意味を検証しているか（実行するだけで検証していないテスト）
- 外部依存（時刻・乱数・ネットワーク・DB）が適切に分離されているか
- フレーキー（実行ごとに結果が変わる）になりやすい書き方をしていないか
- テスト名が「何を確かめるか」を表しているか

## 重要度の目安
- 重要なロジック・バグ修正にテストがない → 🟡 SHOULD（リスクが高ければ 🔴 MUST）
- ケースの不足、境界値の漏れ → 🟢 NITS
- テスト設計の相談 → 💬 QUESTION
