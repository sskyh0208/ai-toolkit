# 設計・スコープ

変更のまとまりと、構造の妥当性を見る。

## PR のスコープ
- 1 PR で 1 関心事になっているか（無関係な変更の混入）
- レビューできない規模に膨らんでいないか（分割の余地）
- リファクタリングと機能変更が混ざっていないか

## 責務・構造
- 変更がレイヤーの責務に沿っているか（ビジネスロジックが適切な層にあるか）
- 既存の抽象・パターンに沿っているか（不要な新パターンの持ち込み）
- 公開インターフェースの変更が妥当か（後方互換・影響範囲）

## 依存
- 依存追加が妥当か（標準・既存で代替できないか、保守性・ライセンス）
- 循環依存・不要な結合を生んでいないか

## 影響範囲
- 破壊的変更・マイグレーション・設定変更が伴うか、その影響が把握されているか
- 機能フラグ・段階的リリースの要否

## 重要度の目安
- 設計上の重大な問題・将来の負債が大きい → 🟡 SHOULD
- より良い構成の提案 → 🟢 NITS
- 設計意図の確認・相談 → 💬 QUESTION
