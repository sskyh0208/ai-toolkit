# PR レビュー

PR の変更内容を読み、**レビュアーとして判断できる指摘**を GitHub にインラインコメントとして投稿する。この skill は2つの skill を束ねるオーケストレーターだ。

- レビューの観点・重要度 → `code-review` skill
- PR の取得・レビュー投稿（GitHub 操作） → `github` skill

この skill 固有の責務は、両者をつなぐ段取り（対象特定 → 取得 → 精査 → 確認 → 投稿）にある。

## 大原則

レビューの正体は**マージ可否の判断材料を渡すこと**だ。指摘の数を増やすことでも、網羅的にコードを読んだ証拠を残すことでもない。

AI にやらせると、ここがずれる。差分を逐次なぞって「ここも直せる」と並べるだけのレビューになる。レビュアーが本当に知りたいのは「このコードは安全にマージできるか」「直すなら何が優先か」だ。指摘は重要度で絞り、マージに影響しない指摘と影響する指摘を混ぜない。

レビュー前に、これに答えられるか確かめる。

- この PR は何を解決しようとしているか（PR 本文・コミットから把握できるか）
- 変更の中で**マージ前に必ず直すべき**箇所はあるか
- 投稿前にユーザーが内容を確認したか

## コアルール

### A. 指摘の質（最優先）

1. **観点と重要度は `code-review` skill に従う**。観点別の精査基準（`references/`）と 🔴 MUST / 🟡 SHOULD / 🟢 NITS / 💬 QUESTION の分類はそちらが本体。
2. **変更されたコードだけを対象にする**。既存コードの問題を新たに指摘しない。
3. **同じパターンは代表1箇所にまとめる**。「同様の箇所が N 件あります」と添える。
4. **MUST がなければ APPROVE する**。指摘がないのに COMMENT にしない。

### B. 投稿前の確認

5. **ユーザーに確認してから投稿する**。指摘一覧を提示し、OK を得てから GitHub に送る。
6. **判定を先に伝える**。APPROVE / COMMENT / REQUEST_CHANGES のどれになるかを最初に示す。

### C. GitHub 操作

7. **取得・投稿は `github` skill に従う**。`gh` CLI による PR の取得（`gh pr view` / `gh pr diff`）、`gh api` の reviews エンドポイントへのインライン投稿、対象リポジトリの特定はそちらが本体。MCP は使わない。
8. **判定（event）を指摘から導く**。MUST が1件以上 → `REQUEST_CHANGES` / それ以外の指摘のみ → `COMMENT` / 指摘なし → `APPROVE`。

## 手順

1. **対象を特定する** — PR 番号（URL なら番号を抽出）と対象リポジトリを確定する。詳細は `github` skill（曖昧なら `-R owner/repo`）。
2. **PR を取得する** — `github` skill の `gh` コマンドで PR 詳細・変更ファイル・diff を取得する（head SHA も把握）。
3. **変更ファイルを読む** — 削除ファイル（`removed`）を除き、diff と実ファイルの両方を確認する。ファイル数が多い場合はビジネスロジック → スキーマ・型定義 → 設定の順で優先する。
4. **レビューを実施する** — `code-review` skill の観点（変更の性質に合う `references/` を読む）で精査し、各指摘に重要度を付ける。
5. **ユーザーに確認する** — 判定予定と指摘一覧を提示し、OK を得る。修正を求められたら指摘を更新して再提示する。
6. **GitHub に投稿する** — `github` skill の `gh api` reviews エンドポイント（`comments` 配列込みの1回 POST）で投稿する。event は重要度から導く（上記 C-8）。
7. **結果を報告する** — 判定・指摘件数・レビュー URL を伝える。

## クイックチェック

投稿前に確認する。

- [ ] 指摘に重要度ラベルを付けているか?
- [ ] 変更されたコードだけを対象にしているか?
- [ ] 同じパターンをまとめているか?
- [ ] MUST がなければ APPROVE にしているか?
- [ ] ユーザーの確認を得てから投稿するか?
- [ ] 取得・投稿を `github` skill の作法で行っているか?

迷ったら優先順位は上から。マージを止める指摘（MUST）の見落としと、確認なしの投稿だけは避ける。

## やらないこと

- 重要度を付けずに指摘を並べる
- 既存コードの問題を新たに指摘する
- ユーザーの確認なしに GitHub に投稿する
- MUST がないのに REQUEST_CHANGES にする
- 指摘がないのに COMMENT にする（APPROVE する）
